What to do if a pod is stuck in Pending state? (commands + debugging)
troductory question
Here’s a **crisp interview-style answer with commands + debugging steps**:

***

**“If a Kubernetes pod is stuck in the `Pending` state, it usually means that the scheduler is unable to place the pod on any node. To debug this, I follow a systematic approach.”**

### 🔍 1. Check Pod Details

```bash
kubectl get pods
kubectl describe pod <pod-name>
```

*   This is the **most important step**.
*   Look at **Events section** for errors like:
    *   `Insufficient CPU/Memory`
    *   `No nodes available`
    *   `PVC not bound`
    *   `node selectors/affinity issues`

***

### 🖥️ 2. Check Node Status

```bash
kubectl get nodes
kubectl describe node <node-name>
```

*   Verify nodes are in **Ready state**
*   Check for:
    *   Resource pressure (CPU, Memory)
    *   Taints blocking scheduling

***

### 💾 3. Check Resource Availability

```bash
kubectl top nodes
kubectl top pods
```

*   If cluster lacks resources → pod stays Pending

***

### 📦 4. Check Persistent Volume Claims (PVC)

```bash
kubectl get pvc
kubectl describe pvc <pvc-name>
```

*   If PVC is **Pending**, pod cannot start

***

### ⚙️ 5. Check Scheduler Constraints

Look for:

*   Node selectors
*   Node affinity / anti-affinity
*   Taints and tolerations

```bash
kubectl describe pod <pod-name>
```

***

### 🌐 6. Check Namespace Quotas / Limits

```bash
kubectl describe quota
kubectl describe limitrange
```

***

### ✅ Common Fixes

*   Increase cluster resources / add nodes
*   Fix PVC or storage class
*   Adjust resource requests/limits
*   Remove incorrect node selectors or tolerations
*   Fix node issues (make nodes Ready)

***

**“In short, I debug a Pending pod by first checking `kubectl describe pod` to identify scheduling issues, then verifying node health, resource availability, storage bindings, and scheduling constraints to quickly pinpoint and resolve the problem.”**
