# ✅ Answer: Configuring Environment Variables & Secrets (Jenkins + GitHub Actions)

**“Environment variables are used for configuration, while secrets are stored securely and injected at runtime. The approach differs slightly in Jenkins and GitHub Actions.”**

***

# 🔷 1. Jenkins Approach

## ✅ Environment Variables in Jenkins

### Option 1: Pipeline (Jenkinsfile)

```groovy
pipeline {
    agent any
    environment {
        APP_ENV = 'production'
        API_URL = 'https://api.example.com'
    }
    stages {
        stage('Build') {
            steps {
                echo "Environment: ${APP_ENV}"
            }
        }
    }
}
```

***

## ✅ Secrets in Jenkins

👉 Jenkins uses **Credentials Plugin**

### Step 1: Store Secret

*   Go to **Jenkins → Manage Jenkins → Credentials**
*   Add:
    *   Username/Password
    *   Secret Text (API key, token)

***

### Step 2: Use Secret in Pipeline

```groovy
pipeline {
    agent any
    stages {
        stage('Use Secret') {
            steps {
                withCredentials([string(credentialsId: 'my-secret-id', variable: 'API_KEY')]) {
                    sh 'echo "Using secret: $API_KEY"'
                }
            }
        }
    }
}
```

***

## ✅ Best Practice in Jenkins

*   Use `withCredentials` (scoped usage)
*   Never echo secrets in logs
*   Use role-based access control

***

# 🔷 2. GitHub Actions Approach

## ✅ Environment Variables

```yaml
name: CI Pipeline

on: [push]

jobs:
  build:
    runs-on: ubuntu-latest

    env:
      APP_ENV: production
      API_URL: https://api.example.com

    steps:
      - name: Print Env
        run: echo "Environment is $APP_ENV"
```

***

## ✅ Secrets in GitHub Actions

### Step 1: Store Secret

*   Go to **Repo → Settings → Secrets → Actions**
*   Add secret:
    *   `DB_PASSWORD`
    *   `API_KEY`

***

### Step 2: Use Secret in Workflow

```yaml
jobs:
  build:
    runs-on: ubuntu-latest

    steps:
      - name: Use secret
        run: echo "Connecting to DB"
        env:
          DB_PASSWORD: ${{ secrets.DB_PASSWORD }}
```

***

## ✅ Environment-Specific Secrets (Advanced)

GitHub supports **environments** (dev, prod):

```yaml
jobs:
  deploy:
    environment: production
```

👉 Each environment can have different secrets

***

# 🔐 Key Differences (Jenkins vs GitHub)

| Feature         | Jenkins                         | GitHub Actions        |
| --------------- | ------------------------------- | --------------------- |
| Env Variables   | `environment {}` in Jenkinsfile | `env:` in YAML        |
| Secrets Storage | Credentials Plugin              | GitHub Secrets        |
| Secret Usage    | `withCredentials`               | `${{ secrets.NAME }}` |
| Scope           | Pipeline-level / step-level     | Job / step level      |

***

# ✅ Best Practices (Important for Interview)

✅ Never:

*   Hardcode secrets in code
*   Print secrets in logs

✅ Always:

*   Use secret managers
*   Use environment-based configs (dev/test/prod)
*   Limit access (least privilege)
*   Rotate secrets regularly
