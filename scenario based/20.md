How to debug an application running in a pod?

Here’s a **strong interview-style answer with practical commands**:

***

**“To debug an application running inside a Kubernetes pod, I follow a structured approach starting from basic inspection to deeper analysis.”**

### 🔍 1. Check Pod Status & Events

```bash
kubectl get pods
kubectl describe pod <pod-name>
```

*   Look for:
    *   Restart counts
    *   Events (crashes, scheduling issues, probe failures)

***

### 📜 2. Check Application Logs

```bash
kubectl logs <pod-name>
kubectl logs <pod-name> -c <container-name>
kubectl logs -f <pod-name>   # follow logs
```

*   Helps identify runtime errors, exceptions, config issues

***

### 🔁 3. Check Previous Container Logs (if restarting)

```bash
kubectl logs <pod-name> --previous
```

*   Useful for **CrashLoopBackOff**

***

### 🖥️ 4. Exec into the Pod

```bash
kubectl exec -it <pod-name> -- /bin/sh
```

*   Inspect inside container:
    *   Environment variables (`env`)
    *   Running processes (`ps`)
    *   Open ports (`netstat` / `ss`)
    *   File system/config files

***

### 🌐 5. Verify Networking

*   Check service connectivity:

```bash
kubectl get svc
kubectl describe svc <service-name>
```

*   Test inside pod:

```bash
curl <service-name>:<port>
```

***

### ❤️ 6. Check Liveness/Readiness Probes

```bash
kubectl describe pod <pod-name>
```

*   Misconfigured probes can restart pods repeatedly

***

### 📊 7. Check Resource Usage

```bash
kubectl top pod <pod-name>
kubectl top nodes
```

*   Look for CPU/memory limits causing crashes

***

### 🧪 8. Debug with Temporary Pod (if needed)

```bash
kubectl run debug-pod --rm -it --image=busybox -- sh
```

*   Helps test DNS/network issues from same cluster

***

### ✅ Interview Closing Line

**“In summary, I start with `kubectl describe` and logs to identify the issue, then exec into the pod for deeper inspection, and finally validate networking, probes, and resource usage to pinpoint the root cause efficiently.”**
