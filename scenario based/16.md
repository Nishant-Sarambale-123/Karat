In Kubernetes, there are **three types of probes** used to monitor container health:

***

# ✅ 1. Liveness Probe

**Purpose:**  
Checks if the container is still alive (not stuck or deadlocked).

### ✅ Success Case

*   The probe succeeds → container is considered healthy
*   Kubernetes **does nothing**, container keeps running normally

👉 Example:
A web app responds with HTTP 200 → OK → container is healthy

***

### ❌ Failure Case

*   Probe fails repeatedly (based on `failureThreshold`)
*   Kubernetes **kills the container and restarts it**

👉 Example:
App is stuck in infinite loop → doesn’t respond → restart triggered

***

# ✅ 2. Readiness Probe

**Purpose:**  
Checks if the container is ready to serve traffic.

***

### ✅ Success Case

*   Probe succeeds → pod is marked **Ready**
*   Kubernetes **adds it to Service endpoints**
*   Traffic starts flowing to the pod

👉 Example:
App has finished startup and DB connection is ready → now accept requests

***

### ❌ Failure Case

*   Probe fails → pod is marked **Not Ready**
*   Kubernetes **removes it from Service endpoints**
*   🚫 No traffic is sent to the pod

👉 Important:

*   Container is **NOT restarted**
*   It just stops receiving traffic

👉 Example:
App temporarily cannot connect to DB → stop traffic but keep running

***

# ✅ 3. Startup Probe

**Purpose:**  
Used for slow-starting applications to prevent premature restarts.

***

### ✅ Success Case

*   Probe succeeds → application has started successfully
*   Kubernetes **switches to liveness + readiness probes**

👉 Example:
Java/Spring Boot app takes 60s to start → startup probe ensures patience

***

### ❌ Failure Case

*   Probe fails → Kubernetes **kills and restarts container**

👉 Example:
App never starts (crash during boot) → restart triggered

***

# ✅ Quick Comparison Table

| Probe Type    | Success Behavior                  | Failure Behavior                      |
| ------------- | --------------------------------- | ------------------------------------- |
| **Liveness**  | Container continues running       | Container restarted                   |
| **Readiness** | Pod receives traffic              | Pod removed from service (no restart) |
| **Startup**   | Enables liveness/readiness probes | Container restarted                   |

***

# ✅ Example YAML

```yaml
livenessProbe:
  httpGet:
    path: /health
    port: 8080
  initialDelaySeconds: 10
  periodSeconds: 5

readinessProbe:
  httpGet:
    path: /ready
    port: 8080
  initialDelaySeconds: 5
  periodSeconds: 3

startupProbe:
  httpGet:
    path: /startup
    port: 8080
  failureThreshold: 30
  periodSeconds: 10
```

***

# ✅ Interview-Ready Answer (Crisp)

> Kubernetes provides three probes: liveness, readiness, and startup. Liveness checks if a container is alive and restarts it on failure. Readiness checks if the application is ready to serve traffic—if it fails, the pod is removed from service endpoints but not restarted. Startup probe is used for slow-starting applications; it delays liveness checks until the app has started, and if it fails, the container is restarted.
