How to design Terraform for multi-environments?

To design Terraform for multiple environments (like dev, test, prod), the goal is to keep the code **reusable, scalable, and clean**.

A common approach is to use **separate environment folders with shared modules**. You create reusable **modules** (for VPC, EC2, database, etc.) and then call these modules in different environment folders like `dev`, `staging`, and `prod`. Each environment has its own **variable files (terraform.tfvars)** to customize things like instance size, region, or scaling.

Another important practice is to use **separate state files** for each environment (often stored remotely in S3 with state locking via DynamoDB). This keeps environments isolated and prevents conflicts.

You can also use **workspaces** for simpler setups, but for production-level systems, directory-based separation is preferred because it’s clearer and easier to manage.

**Example structure:**

    terraform/
      modules/
        vpc/
        ec2/
      environments/
        dev/
          main.tf
          terraform.tfvars
        prod/
          main.tf
          terraform.tfvars

In short, you ensure multi-environment support by **using reusable modules, environment-specific configurations, and isolated state management**, which keeps infrastructure consistent and easy to maintain.
