How to check CrashLoopBackOff and fix it?

Here’s a **strong interview-style answer with commands + fix steps**:

***

**“CrashLoopBackOff indicates that a container in a pod is repeatedly failing to start and Kubernetes is backing off before restarting it. To debug and fix it, I follow a systematic approach.”**

***

### 🔍 1. Identify the Pod

```bash
kubectl get pods
```

*   Look for status: `CrashLoopBackOff`

***

### 📜 2. Check Pod Details (Key Step)

```bash
kubectl describe pod <pod-name>
```

*   Check **Events section** for:
    *   Errors
    *   Exit codes
    *   Probe failures

***

### 📄 3. Check Logs

```bash
kubectl logs <pod-name>
```

👉 If container restarts frequently:

```bash
kubectl logs <pod-name> --previous
```

*   Helps capture the logs before crash

***

### 🖥️ 4. Exec into Container (if possible)

```bash
kubectl exec -it <pod-name> -- /bin/sh
```

*   Inspect:
    *   Config files
    *   Environment variables
    *   Dependencies

***

### 🔍 5. Common Causes & Fixes

#### ✅ Application Crash

*   Issue: Code error / missing dependency
*   Fix: Check logs → fix code/config → redeploy

***

#### ✅ Wrong Command / Entrypoint

*   Issue: Container exits immediately
*   Fix: Verify Dockerfile CMD/ENTRYPOINT

***

#### ✅ Resource Issues (OOMKilled)

```bash
kubectl describe pod <pod-name>
```

*   Look for `OOMKilled`
*   Fix: Increase memory limits

***

#### ✅ Liveness/Readiness Probe Failure

*   Issue: Incorrect health check
*   Fix:
    *   Correct endpoint/port
    *   Increase timeout/initial delay

***

#### ✅ Config/Secrets Missing

*   Issue: App fails due to missing values
*   Fix:
    *   Check ConfigMaps/Secrets

```bash
kubectl get configmap
kubectl get secrets
```

***

### 📊 6. Check Resource Usage

```bash
kubectl top pod <pod-name>
```

***
