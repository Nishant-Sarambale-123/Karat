5.	What is diff between stateful and stateless applications?

In **Kubernetes**, stateful and stateless applications are handled using different types of resources:

A **stateless application** does not store any data in the pod itself. Each request is independent, and any pod can handle it. In Kubernetes, these are usually deployed using **Deployments**. If a pod fails, Kubernetes simply creates a new one without worrying about data. Examples include APIs, frontend apps, or microservices. They are **easy to scale and manage**.

A **stateful application** stores data and needs to maintain identity between restarts. In Kubernetes, these are managed using **StatefulSets**. Each pod has a **stable identity, fixed hostname, and persistent storage (Persistent Volumes)**. Even if a pod is rescheduled, it keeps its data. Examples include databases like MySQL, MongoDB, etc.

**In short (Kubernetes way):**

*   **Stateless → Deployment → no data stored, easy scaling**
*   **Stateful → StatefulSet → persistent data, stable identity**



**Stateful and stateless applications** differ based on how they handle data and user sessions.

A **stateless application** does not store any client data between requests. Each request is independent, and the server treats every request as new. Examples include REST APIs. These applications are **easy to scale, reliable, and simple to manage** because any server can handle any request.

A **stateful application**, on the other hand, **stores data or session information** between requests. This means the server remembers previous interactions. Examples include databases and traditional web apps with user sessions. These are **harder to scale and manage** because the state must be maintained and synchronized.

**In short:**

*   Stateless = No memory, each request is independent (easy to scale)
*   Stateful = Maintains session/data (more complex, needs careful handling)
