The current architecture is deployed on Kubernetes with supporting infrastructure components shown in the diagram below.
What issues do you see in this system?
***Infrastructure Specifications:***
![Diagram](https://public.karat.io/content/devops_system_design_f6x_640bg.png)

Webhook Receiver:
* 2 replicas processing GitHub/GitLab webhooks
* Timeout: 10 seconds per webhook
* Retry failed deliveries every 30 seconds
* Currently handling 100 repositories

Build Controller:
* 1 replica orchestrating all pipelines
* Schedules builds on executors
* Health check interval: 60 seconds
* Maintains build queue in memory

Build Queue:
* Current depth: 150 pending jobs
* Average build duration: 30 minutes
* Priority levels: high, normal, low
* FIFO within each priority level

Build Executors:
* 20 pods with 4GB RAM each
* Namespace quota: 50GB total memory
* Each executor handles 1 build at a time
* Average 2 builds/hour per executor
* Node selector: requires "build=true" label

Pipeline Database:
* Stores build logs and metadata
* Current size: 500GB
* Growth rate: 20GB/day
* 30-day retention policy
* Single writer instance

Secrets Vault:
* 1 replica storing credentials
* Cache TTL: 5 minutes
* Contains cloud provider tokens
* Git tokens expire every 90 days
* Serves 1000 requests/minute

Build Cache:
* StatefulSet with 3 nodes
* 500GB total capacity across nodes
* Current usage: 450GB
* LRU (Least Recently Used) eviction when full
* Network bandwidth: 1Gbps between nodes

Artifact Registry:
* Upload bandwidth: 100Mbps
* Storage capacity: 1TB
* Current usage: 800GB
* Artifacts average 500MB each
* 14-day retention policy
* Growth rate: 100GB/day

Notification Service:
* 5 worker pods sending webhooks
* Processes 50 notifications/second
* Webhook timeout: 5 seconds
* No retry on failure
* Queue depth: 10,000 messages

Kubernetes Configuration:
* Build namespace memory quota: 50GB
* Namespace PVC limit: 100, current PVC usage: 95
* PVC cleanup: Manual process
* 15 nodes with "build=true" label
* 5 nodes with GPU support
* ML builds require GPU nodes

Monitoring:
* Metrics scraped every 30 seconds
* Logs shipped to central storage
* 7-day metric retention

Known Issues:
* Build queue growing throughout day
* "PVC quota exceeded" errors weekly
* Artifact uploads fail for 2GB+ files
* ML builds wait hours for GPU nodes

How would you incrementally fix the system, without it going offline or switching completely to a new system at once?


Below is a **structured system design review + incremental fix plan**. I’ll first identify **key issues (by layer)**, then propose **practical, incremental fixes** that can be rolled out without downtime.

***

# 🔴 1. Critical Bottlenecks & Issues

## 1. Webhook Receiver

**Issues:**

* Only **2 replicas** → low fault tolerance and burst capacity
* **10 sec timeout + retry every 30 sec** → duplicate deliveries under load
* No buffering → directly pushes pressure downstream

**Impact:**

* Drops or delays incoming events
* Cascading overload to build controller

***

## 2. Build Controller (Single Point of Failure 🚨)

**Issues:**

* Only **1 replica**
* Queue stored **in memory**
* 60-second health check → slow failure detection

**Impact:**

* If pod dies → **build queue lost**
* Cannot scale
* Bottleneck for entire system

***

## 3. Build Queue

**Issues:**

* Queue depth = **150 pending jobs**
* Avg build time = **30 min**
* Capacity:
  * 20 executors × 2 builds/hour = 40 builds/hour
  * Incoming > processing → backlog grows
* FIFO within priority → possible starvation of lower priorities

**Impact:**

* Increasing latency for builds
* Poor prioritization fairness

***

## 4. Build Executors

**Issues:**

* **Hard scheduling constraint: `build=true`**
* 15 nodes only → limited scale
* Each executor: **only 1 build**
* Namespace quota: **50GB memory (tight!)**
* ML builds compete for **only 5 GPU nodes**

**Impact:**

* Low utilization flexibility
* GPU starvation (already observed)
* Memory throttling

***

## 5. Pipeline DB (Scaling + Durability Risk)

**Issues:**

* **Single writer instance**
* DB size = **500GB, growth 20GB/day**
* Likely no partitioning

**Impact:**

* Write bottleneck
* Scaling risk
* Future outages due to storage

***

## 6. Secrets Vault (High Risk 🚨)

**Issues:**

* Only **1 replica**
* Serving **1000 req/min**
* Tokens expire every 90 days (manual rotation risk)

**Impact:**

* Single point of failure
* Latency spikes
* Potential outage during restart

***

## 7. Build Cache

**Issues:**

* 500GB total, **450GB used**
* LRU eviction → cache churn
* Network bandwidth: **1 Gbps** (low for CI workloads)

**Impact:**

* Cache misses increase build time
* Network becomes bottleneck

***

## 8. Artifact Registry

**Issues:**

* Upload bandwidth = **100 Mbps** (very low)
* Fails for >2GB files (already observed)
* Growth = **100GB/day**, current = 800GB/1TB

**Impact:**

* Imminent storage exhaustion (\~2 days!)
* Failed uploads → broken pipelines

***

## 9. Notification Service

**Issues:**

* No retries ❌
* Queue depth = **10,000**
* Timeout = 5 sec

**Impact:**

* Message loss
* Delayed notifications

***

## 10. Kubernetes Limits

**Issues:**

* PVC limit: **100 (95 used)** → near exhaustion
* Manual cleanup ❌
* Fixed node label constraints

**Impact:**

* “PVC quota exceeded” errors (already happening)
* Operational overhead

***

## 11. Monitoring

**Issues:**

* Metrics retention = **7 days only**
* Scrape interval = 30 sec (ok but coarse for spikes)

**Impact:**

* Poor capacity planning
* Hard to detect trends

***

# ✅ 2. Incremental Fix Plan (No Downtime)

## Phase 1: Stabilize System (Immediate)

### ✅ 1. Introduce Durable Queue (CRITICAL)

Replace in-memory queue with:

* Kafka / RabbitMQ / AWS SQS

**Steps:**

* Dual-write (memory + queue)
* Gradually switch reads to queue
* Remove in-memory queue

**Benefits:**

* Persistence ✅
* Backpressure ✅
* Decoupling ✅

***

### ✅ 2. Scale Webhook Layer

* Add **HPA (Horizontal Pod Autoscaler)**
* Scale from 2 → 10+ replicas

Add:

* Rate limiting
* Deduplication (idempotency keys)

***

### ✅ 3. Make Build Controller HA

* Run **multiple replicas (3)**
* Use leader election (Kubernetes lease)

***

### ✅ 4. Fix Artifact Failures

* Increase upload bandwidth (100 Mbps → 1 Gbps)
* Add:
  * Multipart upload support
  * Chunked uploads
* Temporary fix: enforce artifact size limits

***

## Phase 2: Remove Bottlenecks

### ✅ 5. Executor Scaling Strategy

* Enable **cluster autoscaler**
* Remove strict `build=true` constraint where possible
* Use:
  * Node pools for general workloads
  * Separate GPU autoscaling group

Enable:

* Multiple containers per executor (if safe)

***

### ✅ 6. GPU Scheduling Fix (Known Issue)

* Dedicated GPU queue
* Pre-warm GPU nodes
* Use:
  * Priority classes
  * Taints/tolerations for isolation

***

### ✅ 7. Fix PVC Exhaustion

* Automate cleanup:
  * TTL controller for PVCs
* Increase quota
* Use ephemeral volumes for short-lived builds

***

### ✅ 8. Expand Build Cache

* Increase size (500GB → 1–2TB)
* Move to distributed cache (e.g., Redis + S3 hybrid)
* Improve eviction (frequency-based instead of LRU)

***

## Phase 3: Data Layer Improvements

### ✅ 9. Scale Pipeline DB

* Add:
  * Read replicas
  * Partitioning (by date/repo)
* Archive old logs → object storage
* Reduce retention (30 → 14 days if acceptable)

***

### ✅ 10. Improve Secrets Vault

* Scale to **3 replicas**
* Add caching layer (sidecar or client-side)
* Automate token rotation

***

## Phase 4: Reliability & Observability

### ✅ 11. Notification Service Fix

* Add retry queue with DLQ (Dead Letter Queue)
* Increase worker count
* Add exponential backoff

***

### ✅ 12. Monitoring Improvements

* Increase retention (7 → 30 days)
* Add:
  * Queue lag metrics
  * Executor utilization
  * Build latency percentiles

***

# 📈 Capacity Insight (Root Cause of Queue Growth)

Current:

* Executors = 20
* Throughput = 40 builds/hour
* Avg build time = 30 min

If incoming rate > 40/hour → backlog inevitable

👉 Fix:

* Scale executors dynamically based on queue length

After fixes:

* Auto-scaling infra
